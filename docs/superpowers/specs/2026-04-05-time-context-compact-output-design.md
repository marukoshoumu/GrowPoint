# 過去事業所情報の区別とモニタリング出力のコンパクト化（設計）

**日付**: 2026-04-05  
**ステータス**: 設計レビュー反映済み（実装は未）  
**関連**: [日付・前回モニタリング表示の正規化](./2026-04-05-date-display-design.md)（同意日のプロンプト埋め込み形式と整合させる）

## 背景・問題

顧客より以下のフィードバックがあった:

1. **過去の事業所の作業をグローポイントの作業として取り込まれている印象がある** — 面談で利用者が語った過去の事業所（デジルミ、フロックス等）での経験が、グローポイントの支援記録に区別なく混在している
2. **全体的にもう少し内容がコンパクトにまとまると嬉しい** — モニタリングシート（就労関係）の特記事項が全項目で同一の長文コピペになっており冗長

## 根本原因の追跡（なぜ × 3）

1. **なぜ (1) が起きるか**  
   Stage2 の抽出項目に「いつの・どこでの事実か」が機械的に区別されず、Stage3 が本文化する際に時系列ラベルが付かないため、過去施設のエピソードが現在の支援文脈と同一に読める。

2. **なぜ (1) だけをプロンプトで「注意深く書け」と言うのでは足りないか**  
   抽出 JSON に安定したタグがないと、長文生成時に一貫した表記（例: 「過去：〇〇」）を再現しづらく、モデルや実行回によってブレる。

3. **なぜ (2) のコピペが起きるか（実装事実）**  
   現行の Stage3-B ハードコードプロンプトは、`monitoring_sheet_evidence` を入力にしつつも、「各項目の特記事項は、**項目に1対1で機械的に紐づけるより、職業生活全体の話題を要約してよい」と明示している。これは**同一長文の横展開を助長しうる**ため、コンパクト化方針と**文言上ぶつかる**。よって本設計では Stage3-B のこの原則を**改める**（後述）。

## 方針決定

- **優先度**: 問題1（過去事業所の混入）を優先。問題2（コンパクト化）も関連するため同時に対処
- **過去情報の扱い**: 完全除外ではなく「過去の経験」として明確にラベリングする
- **時系列判定の基準**: 利用者マスターの同意日を利用開始時期の目安とする（マスタ読込・表示は関連設計 [日付表示](./2026-04-05-date-display-design.md) と同じ正規化を前提にする）
- **判断が曖昧な場合のデフォルト**: `current` とする — **理由**: 「現在の支援に関わる事実を落とす」リスクを抑えるため。**ただし**下記「トレードオフ」参照。過去施設を示す語句（「前の事業所」「デジルミでは」等）が**1つでも明確なら `past_facility` を優先**し、デフォルトは「手がかりが皆無のとき」に限定する、と Stage2 プロンプトで狭める
- **アプローチ**: Stage2 で時系列タグ付け（Stage3 はタグ付き抽出 JSON を使う。本文の「過去：」表記は主にプロンプトで指示）

### トレードオフ（自己検証）

| 懸念 | 内容 |
|------|------|
| デフォルト `current` の副作用 | 境界ケースで過去経験が「現在」に寄ると、顧客の (1) に逆効果。**対策**: 上記のとおり「過去施設の手がかりがあれば `past_facility` 優先」をルールで明文化し、few-shot で暗黙の施設名のみの例も含める |
| コンパクト化と情報欠落 | 項目ごとに絞ると、横断的な文脈が薄れる。**対策**: `monitoring_sheet_evidence` の **当該キーの evidence / note** を第一ソースとし、空の項目は空文字のまま（既存指示と整合） |

## 設計

### 1. Stage2スキーマ変更

**全カテゴリ**の各抽出オブジェクト（`cat1_health` … `cat8_uncategorized` の配列要素。`cat3_wishes` / `cat6_staff` は既存の `sub_type` に加える）に、次の2フィールドを追加する:

```json
{
  "id": "c2-001",
  "quote": "前の事業所でイラスト作業やってて...",
  "speaker": "利用者",
  "summary": "以前の事業所でイラスト作業の経験がある",
  "time_context": "past_facility",
  "facility_name": "デジルミ",
  "flags": [],
  "applicable_sections": ["monitoring_4_status"],
  "dialogue_context": null
}
```

- `time_context`: `"current"` | `"past_facility"` | `"general"`
  - `current`: 同意日以降の出来事、またはグローポイントでの出来事と明確にわかるもの
  - `past_facility`: 「前の事業所」「以前は」「〇〇（施設名）では」等の文脈で語られた出来事
  - `general`: 障害特性、服薬、家族状況など時期に依存しない継続的事実
- `facility_name`: `past_facility` の場合のみ施設名を記載（不明なら `"不明"`）。それ以外は `null`

### 2. Stage2プロンプト変更

利用者マスター情報に「同意日」を追加で渡し（テンプレート変数 `consentDate` 等。表示は `formatJapaneseDate` 済み文字列を推奨）、以下の判定基準を指示:

- 「前の事業所」「以前は」「〇〇（施設名）では」等の文脈 → `past_facility`
- 同意日以降の出来事、またはグローポイントでの出来事と明確にわかるもの → `current`
- 障害特性、服薬、家族状況など時期に依存しない継続的事実 → `general`
- 判断に迷う場合は `current`（ただし上記「手がかりがあれば `past_facility` 優先」より後段のルール）

### 3. Stage3-A（記録票）プロンプト変更

- `past_facility` の情報は「（過去：〇〇事業所での経験）」と明記して記載
- `general` の情報はそのまま記載

出力例（「4. 支援の実施状況」）:
> イラスト作業を楽しいと感じている。パソコン作業に意欲がある。（過去：フロックスでフルーツキャップ、整骨院タオル等の作業経験あり。デジルミでイラスト作業の経験あり）

### 4. Stage3-B（モニタリングシート）プロンプト変更

**既存の矛盾する原則を置換する**: 「項目に1対1で機械的に紐づけるより職業生活全体を要約してよい」を**削除または反転**し、次を最優先とする:

- **データソースの優先順**: 構造化 JSON の `monitoring_sheet_evidence` の **該当項目キー**（例: `work_life.attendance`）の `evidence` / `note` を、その行の特記事項の第一根拠とする。そこが空の項目は空欄
- **各項目に該当するエビデンスのみを個別に記載する**（全項目同一コピペを禁止）
- 面談で言及がなかった項目は空欄（既存: `note` は `""`）
- `past_facility` の情報は記録票と同様に区別表記
- `general` は該当項目にのみ記載

出力例（「1. 職業生活」）:

| № | 内容 | 特記事項 |
|---|------|---------|
| 1 | 遅刻・欠勤しない | 利用開始3日目、現時点で遅刻・欠勤なし。偏頭痛による不眠あり、日中の眠気に注意。 |
| 3 | 健康に気を付けた生活 | 偏頭痛が夜間に悪化、眠剤2錠を服用。運動で体が楽になる。 |
| 8 | 作業に積極的に取り組む | イラスト作業を楽しいと感じている。パソコン作業に意欲あり。農作業は身体的に苦手。 |
| 9 | 集中して取り組む | 現在の作業ペースで疲労なし。 |

言及なしの項目（2, 4, 5, 6, 7, 10, 11）は空欄。

### 5. バリデーション（`gas/extract.gs`）

- `validateStage2Output` を拡張し、各カテゴリ要素に `time_context` / `facility_name` が存在することを検証する（値の列挙チェックは緩めでも可。まずは存在と `past_facility` 時の `facility_name` 必須など）

## 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `gas/prompts.gs` | Stage2 JSON スキーマ文字列に `time_context` / `facility_name` 追加、判定基準・置換ルール追記、few-shot 更新、`consentDate` テンプレート変数追加。Stage3-A/B での過去表記と Stage3-B の「1対1」原則の改定 |
| `gas/extract.gs` | `validateStage2Output` に新フィールド検証を追加 |
| Drive プロンプトファイル（設定済みの場合） | `getPromptFromFile_` 経由の Stage2 / Stage3 ファイルも同内容に同期 |

**補足（誤解防止）**: 記録票・モニタシートの**本文**は `generateDocument.gs` の `runStage3A` / `runStage3B` が LLM に依頼しており、**時系列ラベル付与の主戦場はプロンプト**。`generateDocument.gs` を「表記ロジックの実装箇所」と誤解しないこと（プレースホルダ差し込み・既存の整形処理のみの想定）。

## 変更しないもの

- Stage1（文字起こし）: 変更不要
- `main.gs`, `fileManager.gs`: パイプライン構造は変更しない
- 利用者マスターのスキーマ: 「同意日」は既存データから取得可能（列定義は変更しない）

## 非目標・成功指標

- **非目標**: 施設名の公式データベース照合、利用者の職歴の外部検証
- **成功指標（リリース前の目視確認）**: 過去施設のエピソードに「過去：」系の表記が付くこと。モニタリングシートで、言及のない行の `note` が空であること。記録票とモニタの事実が矛盾しないこと
