# 過去事業所情報の区別とモニタリングシート出力コンパクト化

**日付**: 2026-04-05
**ステータス**: 承認済み

## 背景・問題

顧客より以下のフィードバックがあった:

1. **過去の事業所の作業をグローポイントの作業として取り込まれている印象がある** — 面談で利用者が語った過去の事業所（デジルミ、フロックス等）での経験が、グローポイントの支援記録に区別なく混在している
2. **全体的にもう少し内容がコンパクトにまとまると嬉しい** — モニタリングシート（就労関係）の特記事項が全項目で同一の長文コピペになっており冗長

## 方針決定

- **優先度**: 問題1（過去事業所の混入）を優先。問題2（コンパクト化）も関連するため同時に対処
- **過去情報の扱い**: 完全除外ではなく「過去の経験」として明確にラベリングする
- **時系列判定の基準**: 利用者マスターの同意日を利用開始時期の目安とする
- **判断が曖昧な場合**: `current`とする（現在の情報を漏らすリスクの方が大きいため）
- **アプローチ**: Stage2プロンプトで時系列タグ付け（Stage3は判定済みデータを使うだけ）

## 設計

### 1. Stage2スキーマ変更

各抽出項目に2フィールドを追加:

```json
{
  "id": "c2-001",
  "quote": "前の事業所でイラスト作業やってて...",
  "speaker": "利用者",
  "summary": "以前の事業所でイラスト作業の経験がある",
  "time_context": "past_facility",
  "facility_name": "デジルミ",
  "flags": [],
  "applicable_sections": ["monitoring_4_status"],
  "dialogue_context": null
}
```

- `time_context`: `"current"` | `"past_facility"` | `"general"`
  - `current`: 同意日以降の出来事、またはグローポイントでの出来事と明確にわかるもの
  - `past_facility`: 「前の事業所」「以前は」「〇〇（施設名）では」等の文脈で語られた出来事
  - `general`: 障害特性、服薬、家族状況など時期に依存しない継続的事実
- `facility_name`: `past_facility`の場合のみ施設名を記載（不明なら`"不明"`）。それ以外は`null`

### 2. Stage2プロンプト変更

利用者マスター情報に「同意日」を追加で渡し、以下の判定基準を指示:

- 「前の事業所」「以前は」「〇〇（施設名）では」等の文脈 → `past_facility`
- 同意日以降の出来事、またはグローポイントでの出来事と明確にわかるもの → `current`
- 障害特性、服薬、家族状況など時期に依存しない継続的事実 → `general`
- 判断に迷う場合は`current`とする

### 3. Stage3-A（記録票）プロンプト変更

- `past_facility`の情報は「（過去：〇〇事業所での経験）」と明記して記載
- `general`の情報はそのまま記載

出力例（「4. 支援の実施状況」）:
> イラスト作業を楽しいと感じている。パソコン作業に意欲がある。（過去：フロックスでフルーツキャップ、整骨院タオル等の作業経験あり。デジルミでイラスト作業の経験あり）

### 4. Stage3-B（モニタリングシート）プロンプト変更

- **各項目に該当するエビデンスのみを個別に記載する**（全項目同一コピペを廃止）
- 面談で言及がなかった項目は空欄
- `past_facility`の情報は記録票と同様に区別表記
- `general`は該当項目にのみ記載

出力例（「1. 職業生活」）:

| № | 内容 | 特記事項 |
|---|------|---------|
| 1 | 遅刻・欠勤しない | 利用開始3日目、現時点で遅刻・欠勤なし。偏頭痛による不眠あり、日中の眠気に注意。 |
| 3 | 健康に気を付けた生活 | 偏頭痛が夜間に悪化、眠剤2錠を服用。運動で体が楽になる。 |
| 8 | 作業に積極的に取り組む | イラスト作業を楽しいと感じている。パソコン作業に意欲あり。農作業は身体的に苦手。 |
| 9 | 集中して取り組む | 現在の作業ペースで疲労なし。 |

言及なしの項目（2, 4, 5, 6, 7, 10, 11）は空欄。

## 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `gas/prompts.gs` | Stage2スキーマに`time_context`/`facility_name`追加、判定基準ルール追記、few-shot例更新、`consentDate`テンプレート変数追加 |
| `gas/generateDocument.gs` | 記録票: `past_facility`区別表記ロジック。モニタリングシート: 項目ごとエビデンス振り分けロジック |
| Driveプロンプトファイル（設定済みの場合） | `getPromptFromFile_`経由で読み込まれるStage2/Stage3プロンプトファイルも同様の変更。ハードコード版と内容を同期させる |

## 変更しないもの

- Stage1（文字起こし）: 変更不要
- `main.gs`, `fileManager.gs`: パイプライン構造は変更しない
- 利用者マスターのスキーマ: 「同意日」は既存データから取得可能
