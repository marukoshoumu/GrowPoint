# 音声文字起こし品質問題の原因分類と対策の整理（開発者向け）

**日付**: 2026-04-05  
**採用方針**: 案3（パイプライン層 × 症状インデックスのハイブリッド）

## 背景・問題

Growpoint の Stage1（`transcribe.gs`）は、Gemini に音声ファイルを渡し**全文をプレーンテキストで生成**する。純粋な ASR 専用パイプラインではなく、**マルチモーダル LLM の生成**として転写が得られる。

実データでは次のような**性質の異なる問題**が同時に現れうる。

1. **通常の誤認識・取り違え**（固有名詞、方言、福祉用語、聞き取りにくい発話）。
2. **同一・類似フレーズの異常な連続**（実際の会話がそうでないのに、短いセリフの繰り返しが本文の大半を占める）。これは「音が悪い」だけでは説明しにくく、**生成過程の異常**（反復様の挙動）を疑う必要がある。
3. **メタ情報と本文の矛盾**（例: ヘッダの「低確信度箇所数」と、本文の異常さが整合しない）。

「すべてが正しく文字起こしされなかった」**単一の根本原因**には還元できない。**層ごとの要因**と**症状からの逆引き**の両方で整理する。

## 目標

1. 失敗を**分類**し、疑うべき**パイプライン上の層**と、将来検討する**対策カテゴリ**（実装は別フェーズ）を対応付ける。
2. インシデント時に**症状から**参照できる短い**インデックス**を提供する。
3. 別フェーズでの**優先度付け**に使える判断軸を置く（数値 SLA は本書では定めない）。

## 非目標

- GAS・API の具体的変更内容、パラメータ値、PR レベルの仕様（別タスク）。
- 現場向けの録音マニュアル本体（必要なら別文書）。本書は開発者向けを主とする。

## 用語

| 用語 | 本書での意味 |
|------|----------------|
| **ASR** | 音声→テキストの変換一般。本システムは専用 ASR エンジンではなく LLM 経由の転写も含む。 |
| **マルチモーダル転写** | 音声に加えテキスト指示（プロンプト）で出力が決まる転写。誤認識に加え**生成としての失敗**がありうる。 |
| **反復様の出力** | 短い文が何度も繰り返され、実際の音声の長さ・内容と明らかに合わないパターン。 |

## 分類フレームワーク

### A. パイプライン層（どこを疑うか）

本リポジトリの Stage1 に対応する範囲で層を切る。

| 層 | 内容 | 例となる要因 |
|----|------|----------------|
| **L1 入力** | 元ファイル、長さ、コーデック、音量、ノイズ、無音区間 | ビットレート低下、遠いマイク、クリップ |
| **L2 転送・格納** | Drive 上のファイル、File API へのアップロード | 形式変換の副作用（別経路で再エンコードした場合など） |
| **L3 モデル推論** | `callGeminiApi` による `generateContent`、モデル選定 | 長尺一括処理、不確実区間での**反復様の生成** |
| **L4 プロンプト・出力制約** | `getStage1Prompt` / Drive 上の Stage1 プロンプト | 話者ラベル・メタ行の書式、長さ前提の記述 |
| **L5 保存後テキスト** | `.txt` としての保存、以降の Stage2 への入力 | 異常な長さ・パターンの検知は未実装ならここで初めて可視化 |

層は**排他的ではない**（一つの症状に複数層が関与しうる）。

### B. 失敗モード一覧（層との対応）

| ID | 失敗モード | 観察指標 | 主に疑う層 | 対策カテゴリ（将来） |
|----|------------|----------|------------|----------------------|
| F1 | 固有名詞・用語の誤変換 | 文脈と合わない語、施設名・人名の取り違え | L1, L3, L4 | 用語集・プロンプト、音声改善 |
| F2 | 方言・発話特性による誤認識 | 方言が標準語に寄る、フィラー過多の扱い | L1, L3, L4 | プロンプト（方言の扱い）、モデル |
| F3 | **反復様の異常出力** | 同一～類似の短い行が**極端に**続く、録音時間と内容の密度が合わない | L1, **L3**, L4 | 分割・チャンク、再試行、検知して失敗扱い、モデル／設定の見直し |
| F4 | メタ情報の信頼性低下 | 「低確信度○箇所」等と本文の異常さが矛盾 | L3, L4 | メタの出し方の見直し、後処理で検証 |
| F5 | 欠落・途中で途切れる | 後半が空、明らかに短い | L3, L4, L5 | max トークン、分割、ログ |

### C. 症状インデックス（逆引き）

| 症状 | まず疑うこと | 補足 |
|------|----------------|------|
| **同じ2行が何百回も交互に続く** | **F3**（L3 を最優先に疑う）。実音声が本当にそうでないなら生成異常の可能性が高い。 | 該当時刻付近を**人間が聴取**して切り分け。 |
| 固有名詞だけがことごとく違う | F1, L4 の用語集・文脈 | 音声がクリアな区間でも誤るか確認。 |
| ヘッダの品質メタが楽観的 | F4 | メタをそのまま品質保証に使わない。 |
| 34 分なのに中身が極端に短い／長い | F5, F3 | 文字数・トークン・分割の別フェーズで扱う。 |

## 事例の匿名マッピング（参考）

以下は**構造の説明用**であり、特定個人を示す意図はない。

- 長時間にわたる **「掃除／清掃員」に関する短いフレーズの反復** と、支援者側の短い相槌の繰り返しに見えるパターン → **F3** が支配的とみなすのが自然。L1（その区間の音声が不明瞭）と L3（モデルが反復に陥った）の**併存**もありうる。
- 冒頭付近の聞き取りにくい語の誤り → **F1 / F2**。
- 「低確信度箇所数: 1」などと本文の異常さの不一致 → **F4**。

## 優先度付けの基準（将来フェーズ用）

別フェーズで対策を並べるときの**目安**（数値目標は本書では定めない）。

1. **影響範囲**: 全件に効く対策（検知・分割・再試行）か、個別チューニングか。  
2. **検出のしやすさ**: ログやテキストだけで自動検知できるか。  
3. **コスト・リスク**: API コール増、レイテンシ、運用複雑さ。  
4. **切り分けのしやすさ**: 層が明確なものから手を付けやすい。
